# XML Parser Strategy for D01/D02 Post-Generation Gates

## Scope

This document records the parser strategy for future post-generation quality gates for:

- D01 BPMN XML generated by `src/lib/generators/bpmn-generator.ts`
- D02 draw.io XML generated by `src/lib/generators/drawio-service-blueprint-generator.ts`

This task is decision/documentation only. It does not implement parsing, add dependencies, or modify D01/D02 generators.

## Existing Dependency Check

Current `package.json` dependencies do not include a dedicated shared/server-compatible XML parser such as `fast-xml-parser`, `xmldom`, or `xml2js`.

Relevant existing XML-adjacent tooling:

- `bpmn-js` is already installed and used for BPMN preview/modeling in the browser. It should not be treated as a generic XML parser for quality gates.
- `DOMParser` is used in browser/client-side import helpers. It is not a safe universal choice for shared quality-engine code because it is not available in Node.js without a browser-like runtime or adapter.

Conclusion: the current dependency set is not enough for deterministic shared post-generation XML gates.

## Decision

Use a real XML parser behind a small project-owned wrapper for future D01/D02 post-generation gates.

Recommended follow-up direction:

- Add a small pure JavaScript XML parser dependency, for example `fast-xml-parser`, only after explicit approval in a follow-up implementation task.
- Hide the concrete parser behind a local wrapper so gate logic does not depend directly on one library API.
- Keep generator functions unchanged. Gates should inspect generated output after generation.

No dependency is added as part of this task.

## Rejected Approach: Regex-Only XML Validation

Regex-only XML validation is rejected.

Reasons:

- XML structure, nesting, escaping, namespace declarations, and duplicate/missing elements are not reliably validated by regex.
- Regex scans can produce false confidence for malformed BPMN or draw.io XML.
- Future gates need structural checks such as element existence, attributes, references, and child relationships.

Regex or string checks may still be used only as cheap prechecks, for example empty output or maximum length guards. They must not be the primary XML validation mechanism.

## Parser Wrapper Direction

A future implementation should add a wrapper such as:

- `src/lib/xml/xml-parser.ts`, or
- `src/lib/quality-engine/xml-parser.ts`

The wrapper should return a safe parse result, not throw raw parser errors through gate code.

Suggested behavior:

- Accept XML string input.
- Enforce size limits before parsing.
- Return `{ ok: true, document }` for valid parse results.
- Return `{ ok: false, error }` for malformed XML or unsupported content.
- Report safe metadata only. Do not log full generated XML because process data may be sensitive.

The wrapper lets D01 and D02 gates share one parser strategy while keeping gate-specific validation separate.

## D01 BPMN Gate Validation Scope

Future D01 post-generation validation should parse the BPMN XML structurally and check:

- Root `bpmn:definitions` element or namespace-aware equivalent.
- Required BPMN namespaces, including BPMN model, BPMN DI, DC, and DI namespaces.
- Required top-level structures:
  - `bpmn:collaboration`
  - `bpmn:participant`
  - `bpmn:process`
  - `bpmn:laneSet`
  - `bpmndi:BPMNDiagram`
  - `bpmndi:BPMNPlane`
- BPMN task/event/gateway elements match canonical ProcessTask BPMN types.
- `bpmn:sequenceFlow` `sourceRef` and `targetRef` point to existing BPMN element ids.
- BPMN DI shapes and edges exist for generated nodes and sequence flows.

This gate should report blocker/warning findings through the quality gate model without changing `generateBpmnXml`.

## D02 Draw.io Gate Validation Scope

Future D02 post-generation validation should parse draw.io XML structurally and check:

- Root is a diagrams.net-compatible `mxfile` structure.
- Required nested structures exist:
  - `diagram`
  - `mxGraphModel`
  - `root`
  - `mxCell`
- Cell ids are present and not duplicated.
- Phase columns and row labels exist.
- Each ProcessTask card remains represented as three joined boxes:
  - Header: actor
  - Middle: task name, BPMN type, manual/automatic nature
  - Footer: system/app
- Card geometry basics are present so output can be opened in diagrams.net.

This gate should report structural issues without changing `generateServiceBlueprintDrawioXml`.

## Browser Preview Compatibility

Browser preview behavior can continue using existing tools:

- D01 preview can continue using `bpmn-js`.
- Client-only XML import helpers can continue using `DOMParser`.

The post-generation gate strategy should not rely on browser-only APIs unless the gate is explicitly client-only. For shared quality-engine checks and future automated verification, use a server-compatible parser wrapper.
